# STEREO プロジェクト開発ルール

## 1. エージェントの動作ルール
* **プランニングの徹底**: 複雑な変更や機能実装を行う際は、必ず `docs/` 以下または `implementation_plan.md` を作成/更新し、ユーザーの承認を得てからコードを変更すること。
* **検証の徹底**: 変更後はローカル開発サーバー等で動作を確認し、結果を `walkthrough.md` にまとめること。
* **ドキュメントの維持**: 既存の仕様ドキュメント（`docs/` 配下）の内容やコメントを尊重し、不要なコードクローンを作らないこと。

## 2. STEREO 設計原則（3D専用エンジンと多重入れ子設計）
* **関心事の分離 (Separation of Concerns) と多重入れ子境界の維持**:
  * 3Dワールドロビー環境（STEREO）と、内側で動く2Dミニゲーム環境（MONORAL）の境界を明確にし、相互のコードや依存関係が混入・密結合しないようにすること。
  * MONORAL カセットの起動と描画投影は、ホスト側の Adapter 層（`WebMonitorAdapter` 相当）を通じてのみ行い、STEREO のランタイムコア自体に 2D エンジンの具象ロジックを含めないこと。
* **ENGINEv4 拡張命令セットの忠実なシミュレート**:
  * STEREO は 3D 専用 WebGL エンジンとして、C# / ENGINEv4 互換 of API（`pos`, `gps`, `grid`, `db`, `task` など）を WebGL/JS 側で可能な限り忠実にエミュレートすること。
  * Web側でサポートしない高級機能に対しては、曖昧なフォールバックを行わず、マニフェストの `capabilities` による事前検証によって **Fail-Loud（安全な起動拒否および明示的例外スロー）** を徹底すること。
* **仕様ファーストによる一貫性の担保**:
  * 設計・開発の「正本」は `docs/` 配下の仕様書（`stereo-spec.md` 等）である。具象実装から仕様を逆定義するのではなく、仕様先行で合意形成を図り、設計の一貫性を維持すること。

## 3. 3D UI/UX設計と操作制御原則
* **ユーザーの指定範囲を超えた「やりすぎ」の回避（カメラ制御）**:
  * 3D空間内の筐体やオブジェクトの操作時、背景ドラッグ等の一般操作を阻害しないよう、ポインターの対象メッシュ判定を用いてカメラ（OrbitControls 等）のロックは **局所的かつ一時的** に行うこと。
  * ユーザーが意図した操作の自由度（プレイ中であっても背景スワイプでカメラを傾けられること等）を奪わないよう、指定の境界を厳しく守ること。
* **操作感度に起因する誤爆防止とUI制御の分離設計**:
  * モバイル等のタッチデバイスにおいて、「素早い連打や微細な入力が想定される同一の座標・イベントエリア」に、「画面状態を大きく変更する大域的/破壊的な状態遷移アクション」を混在させてはならない。
  * 大域的または不可逆な状態遷移アクション（筐体からの離脱 `PUT BACK` や射出 `EJECT`）は、3Dモデル上のインタラクションから論理的・物理的に分離された専用の静的 2D UI（画面上部の固定ボタン等）に役割を委譲し、誤爆を構造的に遮断すること。
